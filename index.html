医疗agent的实现

agent在医疗场景下的优势：

主动获取信息而非被动回答。 纯LLM只能基于你给它的信息做推理。Agent可以主动去查——通过FHIR API调取患者的完整病历、通过PubMed检索最新文献、通过药物数据库查询药物交互。文章Section 1提到的核心动机就是：静态LLM受限于训练数据，而Agent可以"consult clinical databases or calculators rather than guessing dosages"。

多步迭代而非单次输出。 疾病诊断不是一问一答的事。Agent可以做到"初步判断→发现信息不足→主动建议做检查→拿到结果→更新判断"这种迭代循环。纯LLM做不到这种跨轮次的状态维护和主动行动。

精确计算而非近似估算。 文章Section 2.5.2专门强调了这点，LLM的算术能力很差。在临床中需要计算风险评分、药物剂量、单位换算等，纯LLM的错误率很高。Agent可以调用医学计算器，论文数据显示这能把错误率降低5到13倍。

记忆和经验积累。 纯LLM每次对话都是从零开始。Agent的记忆架构（文章Section 2.4详细讨论的五种记忆类型）使得它可以积累经验——记住成功的诊断路径（episodic memory）、复用有效的查询模式（procedural memory）、根据反馈持续改进（learning memory）。

可追溯和可审计。 纯LLM给出一个答案，你不知道依据是什么。Agent的每一步行动都有记录——调了什么API、查了什么数据库、检索了哪篇文献——形成完整的决策链条，可以被临床医生回溯审查。

常见的应用场景有：
1. 医疗服务优化 (Healthcare Service Optimization)

服务KPI与患者反馈 → 分析痛点和瓶颈 → 检索行业最佳实践 → 生成工作流改进方案 → 输出流程图和建议 → 运营团队执行并反馈 → 持续监控和优化

2. 医患对话模拟 (Doctor-Patient Interaction Simulation)

设定临床场景和脚本参数 → 解析对话目标（问诊、健康教育）→ LLM Agent生成对话（包含Doctor Agent、Patient Agent、Supervisor Agent三类角色）→ 患者可由Agent或真人扮演 → 记录对话并反馈 → 调整对话策略和提示词 → 迭代直到收敛

核心特点是多Agent协作，Supervisor负责监督对话质量。主要用于医学教育和AI训练，如AgentClinic。

3. 自动化生物医学研究 (Automated Biomedical Research)

设定研究目标和假设 → 解析研究问题和关键变量 → 通过RAG检索文献和数据 → 生成实验设计（由Literature Agent、Experiment Agent、Analysis Agent分工）→ 调用实验室工具或代码解释器 → 收集和分析实验数据 → 研究者审查并反馈

这是一个典型的多Agent分工架构，如BioDiscoveryAgent和CRISPR-GPT。

4. 科学文献分析 (Scientific Literature Analysis)

研究主题或查询 → 自动构建检索query → 从PubMed等数据库检索和筛选论文 → 提取关键信息（方法、结果、结论）→ 按章节生成结构化摘要 → 专家审查和修订 → 最终输出文献综述

本质上是自动化systematic review的流程，可将数周的人工工作压缩。

5. 疾病诊断 (Disease Diagnosis)

症状描述和体征 → NER实体识别进行症状-体征匹配 → 检索鉴别诊断列表 → Tree-of-Thought推理进行诊断探索 → 生成带置信度分数的诊断 → 推荐进一步检查 → 根据检查结果更新诊断

关键特点是迭代推理：不确定时推荐检查，拿到结果后更新结论，模拟临床医生的诊断思维过程。

6. 临床决策支持 (Clinical Decision Support)

患者EHR和检查数据 → 提取病史和风险因素 → 通过RAG检索指南和相似病例 → Chain-of-Thought推理生成决策建议 → 格式化输出风险评分和治疗方案 → 医生审查并反馈

这个领域与FHIR API的集成最紧密，成熟度也最高（如MedAgentBench已有标准化benchmark）。

7. 通用系统工作流 (General System Workflow)

接收输入（症状、测试数据、研究目标、服务指标）→ 意图解析与实体提取（诊断、摘要、设计）→ 知识检索（PubMed、EHR知识图谱、工具API）→ 推理与决策（CoT、ToT、ReAct、多步规划）→ 输出生成（报告、建议、协议、对话、优化方案）→ 反馈迭代与人工审查

这是前六个领域的抽象统一框架，论文也指出所有领域共享"输入解析→知识检索→多步推理→输出生成→人工反馈"这个workflow backbone。

其中成熟度和可行性高的是疾病诊断和临床决策支持这两个。

医疗agent记忆系统

从调研结果来看，Agent Hospital 的这种"原始记录 + 蒸馏经验"双库设计是医疗诊断Agent中最典型的长期记忆架构范式。

核心思路：

第一个库：医疗记录库（Medical Record Library）——存储具体的、未经加工的过往病例记录。每次正确诊断后，完整的病例信息（症状、检查、诊断结果等）直接入库。检索时通过病例相似度匹配，找到与当前患者最相关的历史病例作为参考。这属于情景记忆"我曾经见过一个类似的病人，当时是这样处理的"。

第二个库：经验库（Experience Base）——对成功和失败的诊疗过程进行反思和蒸馏，提取出抽象化的推理原则和诊断规则。比如从多个肺炎误诊病例中提炼出"当患者同时出现X和Y症状时，应优先排除Z疾病"这类泛化规则。这是程序性记忆——跨病例抽象出来的"临床直觉"。

所以你可以把它总结为一个谱系：一端是纯原始记录（MedChain-Agent 的病例数据库），另一端是纯蒸馏规则（MedAgentBench v2），Agent Hospital 和 EvoClinician 处于中间，两者都有且互补。

Agent Hospital

MedAgent-Zero 的完整记忆架构

Agent Hospital 的核心训练策略叫 MedAgent-Zero，它的记忆模块由两个库和一个五步演进流程构成。

库一：医疗记录库（Medical Record Library）

这个库存的是具体的、完整的成功病例。当医生 agent 对一个模拟患者做出正确诊断后，整个诊疗过程——包括患者症状描述、问诊对话、检查结果、诊断结论和治疗方案——作为一条完整的医疗记录直接入库，不做任何加工或抽象。

此外，这个库的初始化不是从零开始的。它还包含来自医学文献和教科书的示范病例（exemplar cases），作为冷启动的知识种子。所以库的内容 = 文献示范病例 + 累积的模拟成功病例。

它的作用类似于一个案例数据库：当面对新患者时，agent 检索出与当前病例最相似的历史成功记录，作为 few-shot 示例注入提示词中，让 LLM 参考"以前类似情况是怎么处理的"。

库二：经验库（Experience Base）

这个库存的是从失败中蒸馏出来的抽象规则。整个流程是这样的：

当医生 agent 对一个患者做出错误诊断时，系统不是简单丢弃这个失败案例，而是触发一个反思流程。agent 被要求对比自己的错误输出和正确答案，分析错在哪里，然后提炼出一条可泛化的经验教训——这就是"经验总结"（experience summarization）。

但蒸馏出来的经验不会直接入库。它还需要过一关验证：系统会用这条新经验去重新测试之前做错的那个病例，看看加上这条经验后能否做对。只有验证通过的经验才会被进一步抽象后写入经验库。验证失败的就丢弃。

所以经验库里的每条内容都是经过"失败→反思→蒸馏→验证→抽象"五步流程淬炼过的高质量推理规则，而不是随便总结的教训。

推理时双库协同检索

推理（inference）时，面对一个新患者，agent 同时从两个库检索：

从记录库中取出 Top-K 个最相似的成功病例作为参考示例（论文实验中 K=2），从经验库中取出 Top-K 条最相关的蒸馏经验作为推理指导（同样 K=2）。

两组内容一起注入到 LLM 的提示词中，agent 同时看到"类似的成功案例长什么样"和"处理这类问题应该遵循什么原则"，然后生成诊断。

检索的相似度度量论文没有特别详述，但从上下文推断应该是基于症状/疾病描述的文本相似度。

消融实验结果

论文做了消融，四种配置：

什么都不用（vanilla）：基线水平。只用记录库：有提升，但有限。只用经验库：也有提升，跟单用记录库差不多量级。两个库都用：效果最好，93.06%（GPT-4），超越人类专家的约 87%。

这说明两个库提供的是不同维度的互补信息。

论文还做了经验积累量的实验（Table 4）：固定用 top-2 经验 + top-2 记录，分别测试从 0/2000/4000/6000/8000 个模拟患者累积的经验库效果。结论是经验越多准确率越高，8000 病例的经验库在 GPT-3.5 和 GPT-4 上都是最好的。这验证了经验库的价值随规模增长。的消融实验也证实了这一点——两个库各自独立贡献性能提升，组合后效果最好。

问题：

Synthea

可以，我直接来操作。先把 Synthea 克隆下来看看结构。

Synthea 确实有 Ground Truth，但粒度有限

有什么（可以当 ground truth 的）：

Synthea 的糖尿病模块定义了一套完整的阶梯用药规则，基于 HbA1c 严重程度分级来决定用什么药。具体来说：

HbA1c ≤ 5.7% → 正常，不开药。HbA1c 5.7-6.5% → 前驱糖尿病（severity 0），只开 CarePlan（饮食+运动）。HbA1c 6.5-7.5% → 糖尿病 severity 1，暂不开药。HbA1c 7.5-9.0% → severity 2，开二甲双胍 500mg 缓释片（单药治疗）。HbA1c > 9.0% 首次 → severity 3，加利拉鲁肽 6mg/mL 注射笔（双药治疗）。HbA1c > 9.0% 持续一年以上 → severity 4，再加卡格列净 100mg 口服片（三药治疗）。如果合并糖尿病肾病 → 转胰岛素（先基础胰岛素 Humulin，再转餐时胰岛素 Humalog）。

CarePlan 也有结构化内容：包括糖尿病饮食、运动治疗两个活动，以及 HbA1c < 7%、空腹血糖 < 108 mg/dL、血压 < 140/90 三个目标。

每条记录都有标准的 SNOMED-CT 和 RxNorm 编码，输出为标准 FHIR 资源（MedicationRequest、CarePlan、Condition、Observation）。

缺什么（作为 ground truth 的局限）：

第一，用药逻辑比较粗。真实指南里糖尿病的二线用药选择取决于很多患者因素（有没有心血管疾病、心衰、CKD、肥胖、低血糖风险等），但 Synthea 简化成了固定的阶梯：二甲双胍 → 利拉鲁肽 → 卡格列净 → 胰岛素，没有个体化分支。

第二，剂量是固定的。每种药只有一个剂量（比如二甲双胍只有 500mg），没有滴定逻辑（真实指南是从 500mg 起步逐步加到 2000mg）。

第三，缺少监测和检查推荐。真实指南要求定期查 HbA1c、肾功能、眼底检查等，Synthea 虽然在就诊时会记录 HbA1c，但没有显式地在 CarePlan 里列出推荐的检查项目和频率。

第四，不考虑禁忌症和药物交互。模块里没有"如果 eGFR < 30 不开二甲双胍"这类安全约束，胰岛素的触发条件简化成了只看有没有糖尿病肾病。

对你项目的影响

Synthea 的输出可以作为方向性的 ground truth——评估 agent 有没有在正确的时间开出正确类别的药（比如 severity 2 的患者是不是开了二甲双胍）。但它不够精细，没法评估 agent 的个体化决策能力（比如考虑合并症选择不同的二线药）、剂量调整推理、安全性校验（禁忌症检查）这些更有价值的能力。

所以我的建议是用 Synthea 作为数据基础，但你需要自己增强 ground truth。具体做法是：用 Synthea 生成患者数据（诊断、检验值、病史等作为输入），然后针对你选的几个疾病，基于临床指南手写或用 LLM 辅助生成更精细的标准治疗方案作为评估基准。Synthea 模块的 JSON 逻辑可以作为起点，你在它的基础上增加个体化分支、剂量逻辑和安全约束即可。有 Ground Truth 吗？

Synthea 生成的 FHIR Bundle 中包含 CarePlan、MedicationRequest、Procedure 等资源 Mitre，这意味着Synthea 生成的数据本身就包含了"这个患者实际接受了什么治疗"的完整记录。

具体来说，Synthea 的疾病模块是状态转移机，支持 MedicationOrder、CarePlanStart、Procedure 等临床状态 PubMed Central，每个疾病模块按照临床护理路径定义了患者应该接受的治疗。这些模块是基于临床护理协议、CDC/NIH 等机构的公开疾病发生率和流行率统计数据以及临床专家反馈创建的 Mitre。所以 Synthea 输出的 CarePlan 和 MedicationRequest 就是根据临床指南和统计数据规则生成的"标准治疗路径"。

这是一个天然的 ground truth 来源，但有几个限制：

可以直接用的部分：Synthea 为每个患者生成了完整的 MedicationRequest（开了什么药）、CarePlan（护理计划包含哪些活动）、Procedure（做了什么检查/手术），这些都可以作为你评估 agent 输出的参照。你可以把一个患者的数据分为两部分——把诊断、检验和病史作为输入给 agent，把 CarePlan/MedicationRequest/Procedure 作为 ground truth 来评估 agent 的输出。

限制：Synthea 生成的是基本 FHIR 资源，包含必填字段但很少填充可选字段 Mitre。这意味 CarePlan 比较粗粒度——它会告诉你"这个糖尿病患者的护理计划包含血糖管理和用药"，但可能不会精确到"二甲双胍每日两次每次 500mg"这种具体剂量层面。MedicationRequest 会有药物名称和编码，但剂量细节可能不完整。所以Synthea 适合做方案方向性的验证（开的药对不对、做的检查对不对），但可能不够支撑精确剂量层面的验证。

指南规则引擎

临床指南规则引擎本质上就是把医学临床实践指南（Clinical Practice Guidelines, CPG）从自然语言文本转化为可计算的 if-then 规则，让机器能够自动执行指南推荐。

举个具体例子。ADA（美国糖尿病协会）指南里有一条：

"对于新诊断的 2 型糖尿病患者，如果 HbA1c < 9% 且无心血管疾病史，一线治疗推荐二甲双胍单药，起始剂量 500mg 每日一次，逐步增加至 2000mg/天。"

这条自然语言规则编码成可执行规则就是：

IF diagnosis == "Type 2 Diabetes"
   AND is_new_diagnosis == true
   AND HbA1c < 9.0
   AND has_condition("cardiovascular disease") == false
   AND eGFR >= 30
THEN
   recommend_medication("Metformin")
   initial_dose = 500mg
   frequency = "once daily"
   target_dose = 2000mg/day
   titration = "increase by 500mg every 1-2 weeks"
   monitoring = ["HbA1c every 3 months", "renal function every 6 months"]

这就是规则引擎做的事情——把指南的临床决策逻辑变成程序可以跑的条件-动作对。

实际的技术实现

业界主要有三种实现路线：

第一种是基于 CDS Hooks + CQL 的标准化路线。HL7 定义了一套专门用于临床决策支持的标准框架。CQL（Clinical Quality Language）是一种专门设计的声明式查询语言，可以直接操作 FHIR 资源写规则。你的场景用 FHIR 数据，这条路线是天然适配的。一条 CQL 规则大概长这样：

define "Needs Metformin":
  exists([Condition: "Type 2 Diabetes"]) 
  and not exists([Condition: "Cardiovascular Disease"])
  and Last([Observation: "HbA1c"]).value < 9.0 '%'
  and Last([Observation: "eGFR"]).value >= 30 'mL/min'
  and not exists([MedicationRequest: "Metformin"])

CDS Hooks 则定义了触发时机——什么时候调用这些规则（比如医生打开处方界面时触发用药建议）。CQL 引擎（比如开源的 cql-engine）可以直接对着 FHIR 服务器执行这些规则。

第二种是传统规则引擎路线，用 Drools 或者 Python 的自定义规则系统。把指南编码成一组规则文件，每条规则有条件部分和动作部分，引擎做前向链推理。这种方式更灵活但不如 CQL 与 FHIR 天然集成。

第三种是决策表/决策树路线。把指南简化成结构化的表格，行是条件组合，列是推荐动作。适合分支相对清晰的指南（比如高血压用药阶梯），但难以处理复杂的多条件交叉场景。

回到你的 agent 架构，规则引擎在两个环节发挥作用：

作为 agent 可调用的工具（第三层校验工具中的 validate_against_guidelines）。Agent 生成了一个治疗方案后，调用规则引擎做合规性校验。引擎会逐条检查：这个患者的情况下，agent 推荐的药是不是指南推荐的一线/二线药？剂量在不在合理范围内？有没有遗漏指南要求的必做检查？如果有违反，引擎返回具体是哪条指南规则被违反了，agent 可以据此修正方案。

作为 ground truth 生成器。你用 Synthea 生成患者数据后，Synthea 自带的 CarePlan 比较粗。你可以用规则引擎对每个 Synthea 患者跑一遍，根据患者的诊断、检验值、合并症等条件，自动生成一份精细的标准治疗方案作为 ground truth。这样评估 agent 输出时就有一个基于指南的客观参照。

规则引擎的主要难点不在技术实现，而在规则的编写和维护。临床指南数量庞大，每条指南有很多分支条件，而且指南每年都在更新。完整编码一个疾病的所有指南推荐是大量的人工工作。生成的细粒度标准方案。Agent 的输出可以在两个粒度上分别评估。

常见数据来源与验证方法

该领域常用数据集全景

这些数据集大致分成三类，每类的用途和 ground truth 性质完全不同：

第一类：真实EHR数据库（原始临床记录）

这是最核心的一类，但它们本身不带任何 ground truth label——它们就是原始的电子病历数据，包含患者的诊断、检验、用药、手术等。你想验证什么任务，需要自己定义 ground truth。

数据集 规模 来源 特点
MIMIC-III ~40,000 ICU患者，2001-2012 波士顿Beth Israel医院，单中心 该领域事实标准，使用最广泛，社区资源最丰富，包含诊断/检验/用药/临床笔记/波形数据
MIMIC-IV ~94,000 ICU入院，2008-2019 同上，更新版 扩展了MIMIC-III，增加了ICU以外的ED数据，更现代的时间跨度，最全面的公开EHR数据集
eICU-CRD ~200,000 ICU入院 美国208家医院，唯一的多中心数据集 规模最大，多中心优势但数据稀疏度高，各医院报告标准差异大
AmsterdamUMCdb ~23,000入院 荷兰阿姆斯特丹大学医学中心，单中心 规模最小但患者最危重（ICU死亡率9.9%），生命体征和检验记录频率较高
HiRID ~33,000入院 瑞士伯尔尼大学医院 时间分辨率最高（心率约30次/小时），适合需要高频时序数据的研究
EHRSHOT 6,739患者 斯坦福医学中心 纵向数据，覆盖所有科室（不止ICU），定义了15个few-shot分类任务

关于 ground truth 的关键理解：这些 EHR 数据库没有预定义的"正确治疗方案"标签。数据里记录的是医生实际做了什么（开了什么药、做了什么检查），而不是应该做什么。研究者需要根据自己的研究问题来定义评估基准。常见做法包括：

用实际临床结局作为 ground truth（比如死亡率预测：患者是否死亡就是 label）。用人工标注的 SQL 查询作为 ground truth（EHRAgent 的做法：人写 gold SQL，agent 的查询结果跟 gold SQL 的结果比对）。用人类医生编写的任务和答案作为 ground truth（MedAgentBench 的做法：300个任务由斯坦福医生手写参考答案和验证规则）。

第二类：医学考试/问答数据集（自带标准答案）

这类数据集来自标准化考试或医学文献，天然自带 ground truth，但它们是静态选择题，不涉及与EHR系统的交互。

数据集 规模 内容 Ground Truth
MedQA (USMLE) ~12,700题 美国执照考试风格的临床vignette + 4/5选项 标准答案（选择题）
MedMCQA ~193,000题 印度医学入学考试题，21个科目 标准答案（选择题）
PubMedQA ~1,000题 基于PubMed摘要的yes/no/maybe问答 标准答案
MMLU (医学子集) ~1,800题 覆盖解剖/药理/临床知识等6个医学方向 标准答案
NEJM Case Challenges 数百例 新英格兰医学杂志的诊断谜题 最终诊断

Agent Hospital 做外部验证时用的就是 MedQA subset。这类数据集评估的是医学知识，不是agent与EHR交互的能力。

第三类：合成/构建的Agent评估环境

这类是专门为评估 agent 而构建的，带有明确的任务定义和评判标准。

数据集/环境 规模 构建方式 Ground Truth

对你的项目的核心结论

MIMIC 确实是该领域的黄金标准数据集，但它的优势在于真实性和社区认可度，并不自带你需要的那种 ground truth。对于你的临床决策支持 agent，情况是：

用 MIMIC 的好处：审稿人/读者立刻认可数据质量，真实患者数据比合成数据更有说服力。用 MIMIC 的问题：它记录的是医生实际行为而不是标准治疗方案，不同医生对同一患者可能有不同的合理决策，所以"实际用药记录"不能直接当成 ground truth（可能有次优决策）。而且需要 CITI 认证和数据使用协议才能访问。

用 Synthea 的好处：免费无门槛，可控可重复，自带基于指南的治疗记录作为粗粒度 ground truth，可以任意生成规模。用 Synthea 的问题：合成数据，审稿人可能质疑临床真实性。

各论文的验证方式对比

这几篇核心论文的验证策略可以归纳为四种：

范式一：模拟环境自验证 + 现成Benchmark迁移（Agent Hospital）

Agent Hospital 的验证分两步。内部验证：在模拟医院里用LLM生成的患者跑10,000个病例，每个病例的"正确答案"来自知识库里预设好的疾病信息（症状、检查结果、诊断、治疗方案都是从医学知识库提前定义好的），所以 ground truth 本质上是人工预定义的疾病模板——8种呼吸系统疾病，每种有对应的标准诊断和三种治疗方案。agent 的输出跟这个模板比对，算准确率。外部验证：把在模拟环境里进化后的 agent 直接拿去做 MedQA（USMLE 执照考试题），这里的 ground truth 就是考试的标准答案。

它没有真实临床数据，ground truth 是知识库模板 + 标准化考试题。验证的是"agent 能不能学会跟预设答案一致"。

范式二：真实EHR + Gold SQL查询（EHRAgent）

EHRAgent 用的是 MIMIC-III 和 eICU 这两个真实ICU数据库。验证任务是临床问答——给一个自然语言问题（比如"多少患者做了临时气管切开？"），agent 需要生成 Python 代码查数据库得到答案。Ground truth 是由人工编写的 gold standard SQL 查询的执行结果。评估指标是 success rate（agent 的答案跟 gold SQL 的答案是否一致）。

ground truth 是人写的SQL + 真实数据库执行结果，不涉及医学推理判断，验证的是"agent 能不能正确查询 EHR 数据"。

范式三：FHIR交互环境 + 人工标注任务（MedAgentBench）

MedAgentBench 来自斯坦福，构建了一个基于真实患者数据的 FHIR 兼容环境，100个患者 profile（超过700,000个数据元素）。300个临床任务由人类医生编写，分10个类别。验证方式分两类：query 类任务（GET请求），用 agent 的返回值跟人工写的 reference solution 比对；action 类任务（POST请求），用人工编写的 rule-based sanity checks 验证 POST payload 的正确性。

ground truth 全是人类医生手工标注的，任务和评分标准都由临床专家设计。最好的模型（Claude 3.5 Sonnet v2）也只达到 69.67% success rate。

范式四：多Agent模拟诊疗 + 多维评估（AgentClinic）

AgentClinic 用四个 LLM agent（医生、患者、检查员、主持人）模拟临床诊疗过程。病例来自 MedQA（USMLE）和 NEJM Case Challenges。Ground truth 是原始病例的标准诊断。评估维度不止诊断准确率，还包括患者的 compliance、confidence、follow-up willingness 等以患者为中心的指标。还加入了认知偏见实验。

ground truth 是现成医学考试/病例集的标准答案，创新在于把静态QA变成了动态交互评估。

可以组合用多种验证方式：

  1. Synthea 自带的粗粒度 ground truth（类似 Agent Hospital 的知识库模板）——验证药物方向是否正确
  2. 临床指南规则引擎校验（类似 MedAgentBench 的 rule-based sanity checks）——验证方案合规性
  3. LLM-as-Judge（类似 AgentClinic 的多维评估）——评安全性、完整性、效率等软指标

融入ai的obsidian笔记软件

目前市面上所有的笔记软件在融入ai的场景都几乎没有，或者说使用难度很高。我们可以在obsidian的基础上简化融入像claudecode这种工具集成进来，让用户以最简单丝滑的方式体验ai，打造自己的数字分身，成为自己的私人教师，又或者是thinking partner等等。
目前已有的ai都是以终端或者插件的形式，可以做成就放在主页面内的一个重要功能，随着用户书写的文字context越多，你的ai就会越懂你，甚至能跨越笔记软件的范畴成为你的一个私人管家。想象一下当你的openclaw能够有你生活的很多context，知道你最近在干什么项目，在想什么问题，经历了什么，明天的todolist，这样的话是不是就有打造一个更加智能贴心的私人贾维斯呢。只不过这个贾维斯战衣的贴合人体是通过用户不断记录笔记或者是导入别的平台笔记进来形成的context。
目前我的看法是这个方案难点在于如何让用户不断的记录自己的生活，让用户培养一个随时写笔记的习惯。剩下的别的部分其实反而倒是没那么难。
一个我能想到最直接的使用场景就是那些考验的学生在笔记软件里面无纸化学习，在笔记软件写题目的时候如果遇到不会的题目，是不是还要切换出去看讲解或者问ai，这个时候就可以直接在笔记软件里面形成一个闭环。

我的想法是目前ai的记忆系统常见的主流方案是使用本地化存储，然后再进行rag提取召回，难点在于召回的精度和基于语义的信息无法通过简单的字符串匹配进行召回相关信息。所以在看了obsidian的连接md文件后产生了一个想法就是：如果我们在拿到了用户的聊天记录后，让llm自己对这一串聊天记录的字符进行识别哪些是对话谈论的主体，哪些内容值得创建一个新的连接（例如obsidian的[[]]一样。这样下次用户再次问到了相关问题后，我们只需要查找该问题的主体所连接的聊天记录，就能够在一个小范围能进行召回，从而提高精度。

  1. 症状描述和体征输入：患者描述症状，或从患者的病历库中通过调用工具提取（FHIR)

  2. 患者EHR和检查数据输入：从电子健康记录系统（通常通过FHIR API）获取患者的完整数据
    该过程难点在于，EHR资源一般都是多跳链接的格式数据，Agent需要在FHIR的图结构数据中导航，跨多个资源类型链式查询。MedAgentBench就是专门为这个场景建的benchmark，用100个合成患者和300个临床任务来测试Agent能否正确操作FHIR API。
  3. NER实体识别与匹配：根据得到的信息提取关键信息例如症状，然后调用外部工具进行知识库中的疾病-症状映射进行匹配
  4. 鉴别诊断列表检索：根据匹配结果，从知识库中拉出一组候选疾病（鉴别诊断）
  5. Tree-of-Thought诊断探索：这是核心推理步骤。不同于简单的线性推理，ToT会分支式地同时考虑多个诊断假设，评估每条路径的合理性，逐步排除不太可能的候选
  6. 生成带置信度的诊断：输出不是单一答案，而是带有置信度分数的诊断结果，比如"肺炎 85%，支气管炎 10%"
  7. 推荐进一步检查：当置信度不够高时，Agent会主动建议做哪些检查来区分剩余候选（比如"建议做胸部CT以排除肺结核"）
  8. 根据检查结果更新诊断：新的检查结果回来后，Agent重新推理，更新诊断结论

描述症状

模拟疾病诊断的输入从哪来

疾病诊断的输入是症状描述，那做研究或测试时这些症状描述的来源主要有几种：

第一种：从合成患者数据中生成。 论文详细介绍了Synthea这个工具。Synthea通过概率状态机模拟疾病进程，生成的是完整的FHIR格式病历，里面包含诊断、检验、用药等结构化数据。但Synthea生成的是结构化记录，不是患者口述的症状描述。所以还需要一步从结构化数据到自然语言症状描述的转化。这可以用LLM来做——把FHIR记录中的Condition和Observation数据转化为模拟患者的口语化描述。

第二种：从真实临床数据集中提取。 论文提到的几个benchmark用了这种方式。AgentClinic从MedQA、NEJM病例报告和MIMIC-IV中提取临床场景。MedJourney用的是中文临床对话数据。这些数据集已经包含了自然语言形式的症状描述和最终诊断的对照。

第三种：用LLM直接生成模拟患者。 论文提到的AgentClinic就是这种思路——用一个Patient Agent来扮演患者，根据预设的疾病参数生成符合该疾病表现的症状描述。Doctor Agent来问诊，Patient Agent根据自己的"病情设定"来回答。这种方式的好处是可以模拟完整的交互式问诊过程，而不只是一段静态的症状描述。

实际的常见做法

在当前研究中最常见的是第二种和第三种的结合。先从真实数据集中获取疾病的ground truth（确诊诊断），然后用LLM基于这个ground truth生成患者角色设定，再让诊断Agent与模拟患者交互来测试诊断准确率。

论文Figure 4展示的benchmark构建流水线就是这个思路：Synthea生成合成患者 → 部署到FHIR数据库 → 基于患者数据编写临床任务 → Agent执行 → 评估。其中"编写临床任务"这一步，对疾病诊断场景来说就是把患者的结构化病历转化为诊断任务的输入描述。

NER实体识别与匹配

  • 提取病史和风险因素：从EHR的多个资源中（Condition、Observation、MedicationRequest等）综合提取关键信息如症状等
  • 调用外部工具进行匹配：知识库中的疾病-症状映射进行匹配

症状-疾病匹配的实现方式主要有几种路径，从简单到复杂：

基于医学知识图谱的匹配

这是最直接的方式。医学领域有现成的大型本体和知识库，比如UMLS（统一医学语言系统）、SNOMED CT、ICD编码体系等，它们内部维护了疾病与症状之间的关联关系。

具体流程是：NER提取出症状实体后，先做实体链接，把自由文本中的症状映射到知识库的标准化概念上。比如患者说"脑壳疼"，需要链接到UMLS中的标准概念"Headache (C0018681)"。然后在知识图谱中查询这个概念关联的所有疾病节点，对多个症状的查询结果取交集或加权合并，就得到了候选疾病列表。

这一步的难点在于实体链接——患者的口语化表达、缩写、同义词都需要正确映射到标准术语。

基于嵌入的语义匹配

把症状和疾病都编码成向量，在嵌入空间中计算相似度。论文中提到的RareAgents就是这种思路，它用患者嵌入去检索最相似的历史病例。具体做法是把当前患者的症状集合编码成一个向量，然后在罕见病数据库中做top-k近邻检索，找到症状模式最相似的历史诊断案例。

这种方法的优势是能捕捉模糊的语义相似性，不需要精确的术语匹配。

基于概率模型的匹配

更传统的医学诊断系统用贝叶斯推理。核心思想是：已知各种疾病出现各种症状的条件概率 P(症状|疾病)，通过贝叶斯公式反推 P(疾病|观察到的症状集合)。每个候选疾病都会得到一个后验概率，这也是最终置信度分数的来源之一。

论文中提到的Synthea合成数据生成就用了类似的概率模型——疾病的状态转移概率来自临床文献和流行病学数据。

为什么不直接用LLM自身知识

理论上LLM在预训练时已经吸收了大量医学知识，确实可以直接做症状到疾病的推理。实际上很多简单场景下LLM也确实是这么干的。但论文反复强调不能只依赖LLM内部知识，原因有几个：

幻觉问题是最核心的。 论文Section 6.1.1专门讨论了这点——LLM可能生成"看起来合理但事实上错误"的匹配。比如它可能自信地把某组症状关联到一个根本不存在的疾病，或者给出错误的症状-疾病关联概率。在普通对话中这不严重，但在临床诊断中可能导致误诊。

知识时效性问题。 LLM的知识截止于训练数据，而医学知识持续更新。新的疾病定义、诊断标准变更、新发传染病等都不在LLM的知识范围内。外部知识库（如UMLS）是持续维护更新的。

可验证性问题。 LLM直接输出"这可能是X病"时，你无法追溯它的依据是什么。但如果是从UMLS查出来的关联，每条匹配都有明确的来源和编码，可以被临床医生审查和验证。论文把这归入Trustworthiness维度。

覆盖度问题。 罕见病是典型例子。LLM训练数据中罕见病的语料非常少，内部知识严重不足。论文提到的RareAgents和DeepRare都是专门解决这个问题的——必须借助专门的罕见病数据库。

所以现实中的做法通常是两者结合：用外部工具做结构化的症状-疾病匹配提供候选列表（确保准确性和可追溯性），再让LLM做更高层的综合推理（利用其强大的上下文理解和推理能力）。

Tree-of-Thought诊断探索

标准CoT（链式思考） 的推理是线性的：症状A+B+C → 可能是疾病X → 验证 → 结论。
ToT的推理是分支式的：

具体实现包含三个关键机制：

第一，思考生成（Thought Generation）。 在每个节点，让LLM生成多个可能的推理方向。实现上就是对同一个prompt做多次采样，或者在prompt中明确要求"请给出3种可能的诊断假设及其理由"。每次生成就是树的一个分支。

第二，状态评估（State Evaluation）。 对每个分支进行打分，判断这条推理路径有多大希望通向正确答案。评估方式有两种：一种是让LLM自己打分（"请评估这个诊断假设的合理性，1-10分"），另一种是用外部工具验证（比如拿假设去知识库查是否有支持证据）。

第三，搜索策略（Search Algorithm）。 决定优先探索哪些分支。常用的有两种：BFS（广度优先）DFS（深度优先）如果评估不理想就回溯换另一条。在诊断场景中，BFS更常用，因为需要同时保持多个鉴别诊断假设。

实现逻辑：
初始化: 根节点 = 患者症状描述
重复:
1. 对当前所有叶节点，调用LLM生成子节点(多个推理分支)
2. 对每个新节点，调用LLM或外部工具评估分数
3. 剪枝: 只保留top-k个最有前途的节点
4. 检查终止条件: 是否有节点达到了足够高的置信度
输出: 得分最高的完整推理路径作为诊断结论

与疾病诊断的结合点在于，ToT天然适合鉴别诊断的思维模式。临床医生做诊断时也不是线性思考的，而是同时维持多个假设，每获得一条新证据就更新各假设的可能性，逐步缩小范围。ToT就是在用搜索算法模拟这个过程。论文中提到的DeepRare处理2919种罕见病的诊断时，就是通过这种"自反思诊断循环"在多轮中逐步收窄候选范围的。

推荐进一步检查

各阶段的信息来源

初始阶段：患者描述驱动。 流程的起点确实是非结构化的症状描述，Agent从中提取症状做初步的鉴别诊断。

ToT推理阶段：知识库驱动。 Agent在展开多个假设分支时，主要调用的是疾病知识库、临床指南等外部知识工具来评估各假设的合理性。

验证和深入阶段：这里就转向API和工具了。 当ToT给出了带置信度的候选诊断后，Agent需要进一步收集证据来区分候选，这时候的信息获取方式会变得多样化：

  • 调FHIR API查历史数据：比如ToT给出"高血压危象 0.6，偏头痛 0.4"，Agent会去GET /Observation?code=blood-pressure查历史血压，如果发现长期高血压记录，高血压危象的置信度就会上升
  • 推荐检查并获取结果：Agent建议做头颅CT，检查结果返回后（在模拟环境中这也是通过API或模拟工具获取的），Agent根据结果更新判断
  • 调用计算工具：比如需要计算某个风险评分来辅助判断
  • 继续问诊：回到患者描述这个通道，追问特定的症状细节（"头痛是搏动性的还是压迫性的？"）

  • 所以整个流程是多通道混合的

第1轮：患者描述 → NER提取 → 知识库匹配 → ToT推理
        ↓
       候选诊断（置信度不够高）
        ↓
第2轮：同时启动多个通道
       ├─ FHIR API → 查历史病历、检验结果
       ├─ 追问患者 → 获取更多症状细节  
       ├─ 推荐检查 → 获取新的检查结果
       └─ 知识库 → 检索更针对性的鉴别诊断信息
        ↓
       更新置信度，重新推理
        ↓
第3轮：如果仍不确定，继续迭代...

论文中的DDO系统就体现了这种混合模式——一个Agent负责向患者问诊收集信息（患者描述通道），另一个Agent负责基于已有信息做诊断推理（可以调用各种工具和API）。两个Agent协作，交替在不同信息通道之间切换。

患者EHR和检查数据获取

这一步是临床决策支持区别于疾病诊断的最显著特征——它的数据入口是结构化的EHR系统，而不是患者的口述症状。

Agent通过FHIR API与电子健康记录系统交互。论文Section 3.1详细讨论了这个过程。比如Agent需要了解一个患者的情况，它会发起一系列API调用：

GET /Patient/123                    → 获取基本人口信息（年龄、性别）
GET /Condition?patient=123          → 获取所有诊断（糖尿病、高血压...）
GET /Observation?patient=123&code=glucose  → 获取血糖检验结果
GET /MedicationRequest?patient=123  → 获取当前用药列表
GET /AllergyIntolerance?patient=123 → 获取过敏史

每次API调用返回的是FHIR格式的JSON数据。这里论文Section 2.4.1提出了一个重要问题：FHIR响应中有大量结构性冗余。一个简单的血糖检验结果可能返回几十个字段（编码系统URI、资源元数据、版本标识等），但对临床推理真正有用的只有检验名称、数值、单位和时间戳。这些冗余会占用Agent的上下文窗口并引入噪声。
设计临床数据蒸馏层——在感知阶段就提取出推理相关的字段——是一个重要但尚未充分探索的改进方向。MedAgentBench目前的做法是把原始API响应直接传给LLM，没有做系统性的蒸馏处理。

提取病史和风险因素

拿到原始数据后，Agent需要从中综合提取出对决策有用的关键信息。这不是简单的数据搬运，而是需要跨多个FHIR资源做信息聚合和关联推理。

举个具体例子：判断一个糖尿病患者是否需要调整用药方案。Agent需要综合：

  • Condition资源中的诊断时间和严重程度
  • Observation资源中近3个月的HbA1c趋势（是在好转还是恶化）
  • MedicationRequest中当前用药的剂量和持续时间
  • Observation中的肾功能指标（因为某些降糖药对肾功能有要求）
  • AllergyIntolerance中的药物过敏记录

这一步的难点在于Agent需要理解FHIR资源之间的图状引用关系，正确地链式查询。论文Section 3.1.1强调FHIR的数据模型不是扁平的关系型数据库，而是图结构的——一个Patient引用多个Observation，每个Observation又引用特定的检验代码。Agent必须能正确地遍历这些链接。

检索指南和相似病例

这一步是临床决策支持的核心知识补充环节，通过RAG实现。

指南检索方面，Agent把提取出的患者关键信息作为查询，去检索相关的临床实践指南。比如对上述糖尿病患者，Agent可能检索到ADA（美国糖尿病协会）最新的血糖管理指南，其中规定"HbA1c持续高于7%且当前单药治疗满3个月，建议联合用药"。

相似病例检索方面，利用论文Section 2.4.2讨论的情景记忆机制。Agent在历史病例库中检索与当前患者特征最相似的案例，看那些病例最终采用了什么治疗方案、效果如何。ReflecTool用BM25做检索，RareAgents用嵌入向量做语义检索。

指南提供的是通用的标准化建议，相似病例提供的是个体化的实践经验。两者结合才能给出既符合规范又贴合患者具体情况的决策建议。

Chain-of-Thought推理生成决策建议

与疾病诊断使用Tree-of-Thought不同，临床决策支持主要使用Chain-of-Thought（链式推理）。原因是这一步的推理相对结构化——已经有明确的诊断、明确的患者数据、明确的指南推荐，Agent要做的是把这些信息逻辑地串联起来，推导出具体的行动建议。

一个典型的CoT推理过程：

已知事实：
- 患者：65岁男性，2型糖尿病，诊断3年
- 当前用药：二甲双胍500mg bid，已服用3个月
- 近期HbA1c：8.2%（目标<7%）
- 肾功能：eGFR 72 ml/min（轻度下降）
- 过敏：无已知药物过敏

推理步骤1：评估当前治疗效果
→ HbA1c 8.2%远高于7%目标，当前方案控制不佳

推理步骤2：对照指南
→ ADA指南：单药3个月未达标，应联合第二种降糖药
→ 候选二线药物：SGLT2抑制剂、GLP-1受体激动剂、DPP-4抑制剂

推理步骤3：结合患者个体因素
→ eGFR 72：SGLT2抑制剂在eGFR>45时可用，且对心肾有额外获益
→ 65岁，应考虑低血糖风险：三类候选药物低血糖风险均较低
→ 无相关过敏史

推理步骤4：参考相似病例
→ 历史相似患者中，二甲双胍+SGLT2抑制剂方案的HbA1c下降幅度中位数为1.1%

结论：建议在当前二甲双胍基础上加用SGLT2抑制剂

这个过程中，Agent可能在某些步骤调用外部工具——比如调用医学计算器计算肾功能分期，或者调用药物交互数据库检查两药是否有相互作用。

格式化输出风险评分和治疗方案

Agent需要把推理结论转化为临床医生能直接使用的结构化输出。这包括：

风险评分的量化。 Agent调用医学计算器工具，计算各种标准化评分。比如心血管风险用Framingham评分或ASCVD评分，肺炎严重度用CURB-65评分。论文提到CalcQA框架支持单位自动换算和灵活的工具编排，确保计算的精确性。

治疗方案的结构化呈现。 不是一段自由文本，而是格式化的建议——具体药物名称、剂量、频次、疗程，以及需要监测的指标和复查时间。

置信度和证据标注。 每条建议标注其依据来源（来自哪条指南、哪个相似病例），以及Agent对该建议的置信程度。这是可追溯性的保障。

这一步可能还涉及写回操作。在FHIR环境中，Agent不仅可以读取数据，还可以发起写入请求。比如：

POST /MedicationRequest  → 提交新的用药医嘱（待医生批准）
POST /ServiceRequest     → 提交检查申请（如建议3个月后复查HbA1c）

这些写入操作在当前系统中都需要经过下一步的人工审批。
